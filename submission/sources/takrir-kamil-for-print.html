1. ملخص المشروع

تطبيق Backend (Laravel 13 + Octane/Swoole) لمتجر إلكتروني يتحمل طلبات متزامنة كثيرة. يركز على: منع أخطاء المخزون (Race Condition)، إدارة السعة، الطوابير للمهام الثقيلة، وقياس الأداء تحت 100 مستخدم.

التشغيل: composer octane + php artisan queue:work — حساب تجريبي: demo@highperformance.test / password

ملاحظة للتسليم (حسب محادثة الأستاذ): التقرير يجب أن يكون PDF وليس Markdown. أداة الاختبار غير محددة — استخدمنا php artisan benchmark:stress بدلاً من JMeter.

4. Race Condition — قبل وبعد القفل

الأمر: php artisan concurrency:race-demo 1 --attempts=30

يبدأ المخزون من 10، ويُنفَّذ 30 خصم متوازي. بدون قفل يفقد التحديثات؛ مع lockForUpdate النتيجة صحيحة.

4.1 بدون قفل (UNSAFE)

Stock reset to 10. Running 30 PARALLEL unsafe decrements..
UNSAFE final stock: 5 (expected 0)

النتيجة: 5 بدل 0 — فقدان تحديثات (lost updates).

4.2 مع lockForUpdate (SAFE)

Stock reset. Running 30 PARALLEL safe decrements (lockForUpdate)..
SAFE final stock: 0

النتيجة: 0 — كل الخصومات طُبِّقت بشكل صحيح.

7. Benchmark — نقاط الاختناق

الأمر: php artisan benchmark:bottlenecks

النتيجة: لا اختناق حرج في العينة الحالية. توصيات النظام:

  • تشغيل Octane + Swoole workers لزيادة throughput
  • تشغيل queue workers منفصلين لكل طابور
  • كاش المنتجات على database (ليس octane table للقوائم الكبيرة)

8. AOP — مراقبة التزامن

ConcurrencyAspect يسجّل مدة عمليات: قفل المخزون، الدفع المحاكى، وطلبات HTTP — بدون تكرار كود القياس في كل Action.

9. ماذا تعلمت

  • Race Condition لا يظهر في طلب واحد — يظهر فقط تحت طلبات متوازية؛ الحل: قفل على مستوى الصف.
  • Octane يحتفظ بالحالة بين الطلبات — لا نخزّن كائنات Eloquent في كاش Octane الصغير.
  • فصل الفواتير والإشعارات في Queue يحسّن زمن استجابة إنشاء الطلب.
  • اختبار 100 مستخدم يمكن تنفيذه بأمر Artisan وليس بالضرورة JMeter.
